### Objetivo principal  
Revisar estado del frontend y backend de prueba, detectar problemas de arquitectura.

### Participantes  
Iván Ibarra, Jorge García, Melisa Abdulgani, Álvaro Fernandez, Álvaro Molina, Jhonatan Sánchez, María Ascensión, Arón

### Agenda  
1. Revisión del frontend basado en JavaScript vanilla
2. Revisión arquitectura del backend  
3. Planificación del login con Google Auth
4. Discusión sobre problemas técnicos

### Tecnologias discutidas
- **Frontend:** HTML/CSS/JavaScript vanilla (confirmado como base)
- **Backend:** Spring Boot, estructura de endpoints RESTful
- **Autenticación:** Google OAuth 2.0 (introducción y análisis)
- **Base de datos:** MongoDB Server (confirmada estable)
- **Docker:** Primeras consideraciones sobre contenedorización

### Puntos tratados  
- El frontend de prueba **no cumple completamente los requisitos de la SRS**, aunque funciona. Se acuerda rehacerlo con mejor estructura modular.
- Se revisa el backend de prueba y se detectan **endpoints mal estructurados:**
  - Faltan métodos POST/PUT/DELETE para casos clínicos
  - LoginController muy básico, solo redirige a Google sin lógica
  - Falta integración real con MongoDB (solo estructuras vacías)
  - ArchivoService y ArchivoRepository no están completamente implementados
  
- Se empieza a planificar el login con **Google Auth:**
  - Discusión sobre flujo OAuth 2.0
  - Necesidad de registrar aplicación en Google Cloud Console
  - configuración de redirect URIs
  
- Problemas técnicos importantes:
  - Docker no levantaba en el portátil de Jorge por configuración de virtuales
  - Confusión con variables de entorno (.env no encontrado)
  - Maven no compilaba por dependencias mal especificadas
  - CORS seguía siendo problema en algunas rutas
  
- Discusiones triviales:
  - Debate sobre usar Postman vs Insomnia para testing de API (se elige Postman)
  - Discusión sobre convenciones de nombres en endpoints (RESTful vs camelCase)
  - Consideración sobre versionado de API (v1, v2...) desde inicio

### Decisiones tomadas  
- Rehacer frontend desde cero con estructura HTML modular y separación clara de responsabilidades
- Crear endpoints definitivos y documentados ANTES de avanzar con Google Auth
- Implementar CrudRepository en ArchivoRepository para operaciones de base de datos
- Estandarizar todas las rutas a patrón RESTful: `/api/v1/casos`, `/api/v1/archivos`, etc.
- Usar Postman para testing de endpoints
- Crear archivo modelo (example.env) para variables de entorno

### Riesgos identificados  
- Retraso en la implementación del login por necesidad de reestructura
- Falta de experiencia en OAuth 2.0 → requiere investigación adicional
- Dependencia de Google Cloud Console para integración
- Posible retraso en Docker si no se resuelven requisitos del sistema
